### 问题 3: Asset 去重机制的完整性

**位置**: `models/l1/asset.py`

**当前实现**:
```python
hash: str = Field(max_length=64, index=True, unique=True, ...)
```

**潜在问题**:
1. **哈希冲突**: 虽然 SHA256 冲突概率极低，但理论上存在
2. **删除策略**: 当最后一个引用的 Sample 被删除时，Asset 是否应该删除？
3. **引用计数**: 没有追踪有多少 Sample 引用了该 Asset

**建议方案**:

#### 方案 A: 添加引用计数（简单）
```python
class Asset:
    reference_count: int = Field(default=0, description="Number of samples referencing this asset")
```

**优点**: 实现简单  
**缺点**: 需要应用层维护一致性

#### 方案 B: 定期垃圾回收（推荐）
```python
# 定期任务：清理无引用的 Asset
SELECT a.id 
FROM asset a 
LEFT JOIN sample s ON JSON_CONTAINS(s.asset_group, CONCAT('"', a.hash, '"'))
WHERE s.id IS NULL
AND a.created_at < NOW() - INTERVAL 7 DAY;
```

**优点**: 解耦，无需实时维护  
**缺点**: 有延迟

**优先级**: 🟢 **P3 - 低（可后续优化）**

### 问题 4: CommitAnnotationMap 的数据一致性

**位置**: `models/l2/camap.py`

**当前实现**:
```python
project_id: uuid.UUID = Field(foreign_key="project.id", index=True)
```

**问题描述**:
- `project_id` 是冗余字段（可以通过 commit_id → commit.project_id 获取）
- 但如果 commit.project_id 和 map.project_id 不一致，会导致数据混乱

**建议**:
1. **保留冗余字段**（性能优化）
2. **添加触发器或约束**确保一致性:
   ```sql
   CREATE TRIGGER check_commit_annotation_map_project
   BEFORE INSERT ON commit_annotation_map
   FOR EACH ROW
   BEGIN
       DECLARE commit_project_id UUID;
       SELECT project_id INTO commit_project_id FROM commit WHERE id = NEW.commit_id;
       IF NEW.project_id != commit_project_id THEN
           SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'project_id mismatch';
       END IF;
   END;
   ```
3. **或者移除 project_id**，在查询时 JOIN:
   ```python
   # 移除冗余字段
   class CommitAnnotationMap:
       # project_id: uuid.UUID  # 移除
   ```

**权衡**:
- **保留**: 查询快，但有一致性风险
- **移除**: 查询慢（需要 JOIN），但数据绝对一致

**建议**: 保留，但在应用层严格校验

**优先级**: 🟡 **P2 - 中等**

### 问题 5: Annotation 修改历史查询性能

**位置**: `models/l2/annotation.py`

**当前实现**:
```python
parent_id: uuid.UUID | None = Field(
    default=None,
    foreign_key="annotation.id",
    index=True,  # ✅ 已有索引
    ...
)
```

**潜在问题**:
- 查询完整修改历史需要递归 `parent_id` 链
- 如果修改次数很多（如 10+ 次），性能可能下降

**建议优化**:

#### 方案 A: 添加 root_id 字段
```python
class Annotation:
    parent_id: uuid.UUID | None
    root_id: uuid.UUID = Field(description="ID of the first annotation in the chain")
```

**优点**: 快速查询同一标注的所有版本  
**缺点**: 需要额外存储

#### 方案 B: 使用 PostgreSQL 递归 CTE（推荐）
```sql
WITH RECURSIVE history AS (
  SELECT id, parent_id, data, created_at, 0 as depth
  FROM annotation
  WHERE id = 'target_id'
  
  UNION ALL
  
  SELECT a.id, a.parent_id, a.data, a.created_at, h.depth + 1
  FROM annotation a
  JOIN history h ON a.id = h.parent_id
  WHERE h.depth < 100  -- 防止无限递归
)
SELECT * FROM history ORDER BY depth;
```

**优点**: 不需要额外字段，SQL 标准支持  
**缺点**: 深度过大时性能下降（但通常不会超过 10 层）

**优先级**: 🟢 **P3 - 低（可监控后优化）**

## 💡 设计建议（增强功能）

### 建议 1: 添加 Annotation 的软删除支持

**背景**: 当前 Annotation 不可变，但用户可能需要"删除"标注。

**建议实现**:
```python
class Annotation:
    is_deleted: bool = Field(default=False, index=True)
    deleted_at: datetime | None = None
    deleted_by: uuid.UUID | None = None
```

**查询时过滤**:
```python
annotations = session.query(Annotation).filter(
    Annotation.is_deleted == False
).all()
```

**优先级**: 🟡 **P2 - 中等**

### 建议 2: 为 Commit 添加 tag 支持

**背景**: Git 支持 tag，用于标记重要版本（如 "v1.0-release"）。

**建议实现**:
```python
class CommitTag(SQLModel, table=True):
    __tablename__ = "commit_tag"
    
    commit_id: uuid.UUID = Field(primary_key=True, foreign_key="commit.id")
    tag_name: str = Field(max_length=100, unique=True)
    description: str | None
    created_at: datetime
    creator_id: uuid.UUID | None
```

**用途**:
- 标记重要里程碑（如 "第一轮标注完成"）
- 快速回溯到特定版本

**优先级**: 🟢 **P3 - 低（增强功能）**

### 建议 3: 为 Branch 添加合并支持

**背景**: Git 支持分支合并，但当前设计没有合并机制。

**挑战**:
- 标注冲突如何解决？（同一 Sample 在两个分支有不同标注）
- 合并策略：保留哪个？都保留？提示用户选择？

**建议实现**:
```python
class BranchMerge(SQLModel, table=True):
    __tablename__ = "branch_merge"
    
    source_branch_id: uuid.UUID = Field(foreign_key="branch.id")
    target_branch_id: uuid.UUID = Field(foreign_key="branch.id")
    merge_commit_id: uuid.UUID = Field(foreign_key="commit.id")
    
    conflicts_resolved: Dict[str, Any]  # 记录冲突解决方案
    merged_at: datetime
    merged_by: uuid.UUID
```

**合并策略**:
1. **最新优先**: 取时间戳更晚的标注
2. **人工优先**: 人工标注优先于模型预测
3. **手动解决**: 提示用户选择

**优先级**: 🟢 **P3 - 低（高级功能）**

### 建议 4: 标注“本人可见”作为可选能力（Self Scope）

**背景**: 目前“项目级克隆 + 项目授权”已满足需求，但未来可能需要同一项目内的细粒度隔离。

**建议策略**:
- 先不默认启用，作为可选能力预留
- 仍使用项目级角色绑定，权限通过 `scope=self` 表达
- 查询层在 scope=self 时追加过滤：`Annotation` / `Draft` / `Commit` / `Branch`

**收益**:
- 适合外包/竞赛式标注场景
- 权限模型不新增表，成本可控

**代价**:
- 协作可见性下降，需在产品侧提示

**优先级**: 🟢 **P3 - 低（可选功能）**
