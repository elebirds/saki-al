# Saki Runtime 上手说明书（面向工程师）

> 目标读者：会写代码，但刚接手本项目。
> 范围：不含主动学习算法细节，聚焦 runtime 控制闭环与调试路径。

---

## 1. 你先建立一个正确心智模型

系统分成两端：

1. `saki-api`：控制面
- 管理 loop/job/task 状态
- 派发任务
- 接收事件与结果

2. `saki-executor`：执行面
- 注册能力
- 拉数据并执行插件
- 回传日志/指标/制品/候选样本

一句话：API 决策，Executor 干活。

## 3. 本地启动最短路径

## 3.1 启 API

确保 PostgreSQL、Redis、对象存储配置正确后，启动 API（你的项目已有部署脚本）。

## 3.2 启 executor

在 `saki-executor/.env` 配置：
1. `API_GRPC_TARGET`
2. `INTERNAL_TOKEN`
3. `EXECUTOR_ID`

然后启动 executor。

## 3.3 验证连接

看 API 日志是否出现 executor 注册；看运行时插件目录接口是否能看到插件能力。

---

## 4. 一次最小闭环（建议你第一次就这么验）

1. 创建项目、分支、标签和样本。
2. 创建 loop（选择一个已注册 plugin_id）。
3. start loop。
4. 观察：
- `job` 是否创建
- `job_task` 是否进入 `DISPATCHING/RUNNING`
- `task_event` 是否持续增长
- `task_result` 后 job 是否终态
5. 查询 task artifacts 下载链接是否可用。

---

## 5. 常见排障路径（按现象）

## 5.1 Task 一直 PENDING

排查顺序：
1. executor 是否在线注册（`runtime_executor`）
2. plugin_id 是否匹配
3. task 依赖是否满足（depends_on）
4. dispatcher allowlist 是否误配

## 5.2 Task 卡 DISPATCHING

重点看：
1. ACK 是否收到
2. `_pending_assign` 是否超时回收
3. executor 是否断线

## 5.3 有日志但无结果

检查：
1. executor 是否抛异常并走 failed result
2. runtime_control 是否成功持久化 task_result
3. task_id 是否一致

## 5.4 制品无下载链接

检查：
1. executor 是否拿到 upload ticket
2. artifacts 中 URI 是否是 `s3://` 或 http(s)
3. 对象存储配置、签名有效期

---

## 6. 你改代码时的边界原则（非常重要）

1. 协议层改动只在 `proto + codec` 收口。
2. 编排规则改动优先在 `loop_orchestrator + JobService`。
3. 调度规则改动在 `dispatcher`，不要侵入业务服务。
4. 算法与训练逻辑只进插件，不要放进 manager。

---

## 7. 现阶段已知技术债（你可以直接开 issue）

1. dispatcher 关键状态为进程内内存，多实例一致性不足。
2. Job 聚合逻辑在 orchestrator/control 两处重复。
3. payload 大量 dict，DTO 强类型化仍可推进。
4. runtime_executor_stats 读接口有，但写入链路未明显闭环。

---

## 8. 建议你第一周完成的熟悉任务

1. 跑通一轮 loop 并画一张时序图（基于日志）。
2. 在 executor 插件里加一个新策略名并端到端验证上报。
3. 手动制造 ACK 超时，观察 task 回滚到 pending 的恢复路径。
4. 给任务结果链路补一条断言测试（event + result 一致性）。

---

## 9. 术语速查

1. Loop：实验编排容器
2. Job：一轮执行聚合
3. Task：可派发执行最小单元
4. Dispatcher：按 executor 资源派发 task
5. RuntimeControl：gRPC 流控制面
6. Plugin：模型训练/推理实现

---

## 10. 一句话结束

你在这个项目里最需要守住的是：
- 控制链路稳定性（状态正确、可恢复）
- 插件边界清晰（算法可替换）
- 观测闭环完整（事件、结果、制品可追踪）
