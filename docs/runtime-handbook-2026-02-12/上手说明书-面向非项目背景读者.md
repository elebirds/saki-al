# Saki Runtime 上手说明书（面向非项目背景读者）

> 目标读者：懂一点编程，但完全不懂这个项目。
> 目标：快速理解“系统在做什么、为什么这么分、怎么看是否正常”。

---

## 1. 这个系统是干什么的（先不讲实现）

你可以把它当成“一个会自动训练与筛选样本的工厂”：

1. 你准备数据与标注。
2. 系统按轮次自动做训练、评分、筛选。
3. 每一轮都会留下日志、指标和模型制品。

---

## 2. 两个核心角色

1. 控制中心（API）
- 负责下发任务、记录状态、保存结果。

2. 工人节点（Executor）
- 真正跑训练、做评分、上传结果。

如果类比外卖系统：
- API 像调度平台；
- Executor 像骑手。

---

## 3. 你在界面上看到的 Loop / Job / Task 是什么

1. Loop：一次完整实验流程。
2. Job：Loop 里的某一轮工作包。
3. Task：Job 中可执行的最小步骤（训练、评分、筛选等）。

关系：
- 一个 Loop 有多个 Job；
- 一个 Job 有多个 Task。

---

## 4. 一次任务从开始到结束发生了什么

1. 你在前端创建并启动 Loop。
2. API 创建本轮 Job 和 Task 列表。
3. API 把 Task 派发给可用 Executor。
4. Executor 拉取所需数据，执行插件里的训练/推理逻辑。
5. Executor 持续回传日志与进度。
6. 执行完成后回传最终结果（指标、候选样本、制品）。
7. API 保存结果并决定是否进入下一轮。

---

## 5. 为什么会有“插件”

插件是为了让“模型算法”可以替换。

系统核心（API + Executor 本体）不关心你用 YOLO 还是别的模型，
它只要求插件遵守统一接口：
1. 准备数据
2. 训练
3. 对未标注样本打分
4. 停止任务

这样可以做到：
- 平台稳定，算法可迭代。

---

## 6. 怎么判断系统现在是健康的

看四类信号：

1. 连接信号
- executor 是否在线，是否持续心跳。

2. 执行信号
- task 是否在 `pending -> running -> succeeded/failed` 正常推进。

3. 观测信号
- task 日志、进度、指标是否实时更新。

4. 结果信号
- 最终是否能看到制品下载、指标曲线、候选样本。

---

## 7. 常见“看起来像坏了”的情况

1. 一直 pending
- 常见原因是没有可用 executor 或插件不匹配。

2. 一直 dispatching
- 常见原因是 executor 没有 ACK 或网络中断。

3. 有日志但没结果
- 常见是执行中异常，结果未成功回传。

4. 有结果但没制品
- 常见是对象存储上传失败。

---

## 8. 你只需要记住的 5 个关键点

1. API 管“调度与记录”，Executor 管“实际执行”。
2. Loop/Job/Task 是三级拆分，不是重复概念。
3. 插件决定算法，本体决定稳定性。
4. 日志与结果分开上报，二者都要看。
5. 先看状态机再看算法，排障效率最高。

---

## 9. 你出去看文档时可以重点观察的设计问题

1. 多实例时调度状态是否一致。
2. 任务状态聚合是否只有一套逻辑。
3. payload 是否过多用字典导致隐式错误。
4. 统计数据是否有完整采集-存储链路。

---

## 10. 一句话总结

这个项目不是“一个训练脚本”，而是“一个可持续运行的训练调度系统”。
看懂它的关键不在模型，而在 `状态机 + 调度 + 回传` 三件事。
